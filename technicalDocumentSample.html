    Good code is readable

    Just because your code is in a language you understand, doesn't necessarily mean your code is readable. Good code is easy to look at, meaning it shouldn't strain the eye. It should be intuitive to easily parse, and the person reading your code shouldn't have to spend time trying to separate words or characters.

    • Always use spaces between variables
    • Make good use of indentation
    • Use new lines when appropriate
    • If using similar arrays, align them vertically
    • Cleanliness is key

    Let's take a look at an example.

    function thisFunction(){ var aVariable; var thisArray=[1,2,3,4]; var anotherVariable; var thatArrayWithAVeryLongName=[1,2,3,4]; for (i=0;i<thisArray.length;i++){ thisArray[i]=thatArrayWithAVeryLongName[i];}};

    This code is unpleasant to look at. Let's make some changes by following the guidelines listed above.

    function thisFunction() { var aVariable; var anotherVariable; var thisArray = [1, 2, 3, 4]; var thatArrayWithAVeryLongName= [1, 2, 3, 4]; for (i=0; i<thisArray.length; i++) { thisArray[i]=thatArrayWithAVeryLongName[i];} } };

    This is much better. We've aligned our arrays, added spaces between our variables, and weren't afraid to use newlines and indent our code. It's still not perfect, and that's because...

    Good code is documented

    Just because you know what your code does, doesn't mean other people do. When writing code, you should have comments in your code to let other programmers understand how it works. The more complicated the code, the more useful commenting your code is!

    Let's apply commenting to the above code.

    //This function sets the value of thisArray equal to thatArrayWithAVeryLongName function thisFunction() { var aVariable; var anotherVariable; var thisArray = [1, 2, 3, 4]; var thatArrayWithAVeryLongName= [1, 2, 3, 4]; for (i=0; i<thisArray.length; i++) //This for loop iterates through the length of thisArray, updating each value at a time. { thisArray[i]=thatArrayWithAVeryLongName[i];} }; }

    This is a lot better, and a new programmer can look at your code and instantly know what it does. There's still more we can do, and that's because...

    Good code is consistent

    You may notice that some things in our above code don't always look the same. You should always strive for consistency in your code, regardless of how you decide to format it. Let's try making our above code more consistent.

    //This function sets the value of thisArray equal to thatArrayWithAVeryLongName function thisFunction() { var aVariable; var anotherVariable; var thisArray = [1, 2, 3, 4]; var thatArrayWithAVeryLongName= [1, 2, 3, 4]; //This for loop iterates through the length of thisArray, updating each value at a time. for (i=0; i<thisArray.length; i++) { thisArray[i]=thatArrayWithAVeryLongName[i]; } };

    What did we do here? You may notice that now, brackets appear consistently throughout our code. Brackets always have their own line, and appear with the same indentation as the function or loop that spawned them. We also now have all of our commenting directly above the line it is explaining. We can make this even better, though...

    Good code follows conventions

    There are pre-existing naming conventions for variables, and they vary for each different programming language. Given that our imaginary code is in JavaScript, we will be going over the standard naming conventions for JavaScript variables.

    Classes Nouns in UpperCamelCase, with the first letter of every word being capitalized.
    Methods Verbs in lowerCamelCase, with the first letter of the verb being lowercase.
    Variables Written in lowerCamelCase. They should be short and relevant.
    Constants Constants are written in uppercase, separated by underscores.

    Let's apply these standards to our code.

    //This function sets the value of firstArray equal to secondArray function arrayIterator() { var firstVariable; var secondVariable; var firstArray = [1, 2, 3, 4]; var secondArray = [1, 2, 3, 4]; //This for loop iterates through the length of firstArray, updating each value at a time. for (i=0; i < firstArray.length; i++) { firstArray[i] = secondArray[i]; } };

    This code makes more sense. We renamed our variables, which allows for our arrays to line up better. We also remembered to update our commenting to reflect the new names of the variables. But we're still not done, because...

    Good code is never done

    Good code is simply never done. When your code works, that is the beginning, not the end of the improvement process. You should always strive to improve on your work, refactoring it to work as efficiently as possible. Let's take a look at one example.

    //This function sets the value of firstArray equal to secondArray function arrayIterator(firstArray, secondArray) { //This for loop iterates through the length of firstArray, updating each value at a time. for (i=0; i < firstArray.length; i++) { firstArray[i] = secondArray[i]; } };

    Here, we removed firstVariable and secondVariable, as they added nothing to the code. Secondly, we changed the code so that arrayIterator accepts two arrays as arguments, rather than having our arrays hardcoded into the function, allowing us for greater flexibility of use.
